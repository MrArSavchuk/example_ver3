# Architectural Decisions

Этот документ фиксирует ключевые архитектурные решения проекта  
**Olesya Martin Designer**.

Цель:
- сохранить контекст
- понимать, *почему* было принято то или иное решение
- избежать повторных обсуждений одного и того же

---

## 1. Feature-Sliced Design (FSD)

**Статус:** принято  
**Дата:** 2026  

### Контекст

Проект предполагает рост:
- по функциональности
- по количеству компонентов
- по сложности логики

Хранить код «по папкам components / hooks / utils» перестаёт быть удобно.

### Решение

Использовать **Feature-Sliced Design** как основную архитектуру фронтенда.

**Почему:**
- чёткие границы ответственности
- изоляция фич
- проще масштабировать и поддерживать
- легче читать код спустя время

### Последствия

- требуется дисциплина при размещении файлов
- иногда путь до файла длиннее, но смысл — яснее

---

## 2. Mobile First

**Статус:** принято  
**Дата:** 2026  

### Контекст

Основная аудитория проекта взаимодействует с сайтом с мобильных устройств.  
Проект — визуальный, дизайнерский, с акцентом на UX.

### Решение

Использовать подход **Mobile First**:
- дизайн
- верстка
- логика адаптации

**Почему:**
- проще масштабировать интерфейс вверх
- меньше костылей с media queries
- UX на мобильных — приоритетный

### Последствия

- дизайн и UI всегда проверяются сначала на мобильных
- desktop — это расширение, а не основа

## Централизация UI-библиотек

**Статус:** принято  
**Дата:** 2026  

### Контекст

Проект использует сторонние UI-библиотеки
для реализации сложных интерфейсных элементов
(уведомления, слайдеры, пагинация и т.д.).

Прямое использование библиотек в фичах приводит к:
- дублированию логики
- разному UX
- сложной замене решений в будущем

### Решение

Все сторонние UI-библиотеки используются
через собственные обёртки в слое `shared/ui`.

Каждый UI-паттерн имеет:
- единую точку входа
- централизованные настройки
- проектную документацию в `docs/ui`

### Последствия

- UI-поведение становится предсказуемым
- библиотеку можно заменить без массовых правок
- новые разработчики используют готовые решения

Проект использует иконки как часть пользовательского интерфейса

## Интернационализация (i18n)

**Статус:** принято  
**Дата:** 2026  

### Контекст

Проект изначально развивается с поддержкой **двух языков**  
и ориентирован на многоязычную аудиторию.

Использование текстов напрямую в UI:
- усложняет поддержку
- делает изменения дорогими
- создаёт риск расхождений в интерфейсе

### Решение

Использовать централизованную систему **i18n**
для всех пользовательских текстов проекта.

Язык рассматривается как часть состояния приложения,
а тексты — как отдельный слой ответственности.

### Почему

- упрощается поддержка нескольких языков
- изменения текстов не затрагивают UI-логику
- интерфейс остаётся согласованным
- проект готов к масштабированию без рефакторинга

### Последствия

- добавление любого текста требует дисциплины
- переводы становятся частью архитектурного контракта
- ошибки в переводах выявляются на ранних этапах

Это осознанное решение в пользу предсказуемости и долгосрочной поддержки.

## Использование общей библиотеки React-хуков

**Статус:** принято  
**Дата:** 2026  

### Контекст

В проекте регулярно возникает повторяющаяся логика:
работа с состоянием, событиями, браузерными API и UI-паттернами.

Копирование одинаковых хуков между проектами
приводит к расхождениям и усложняет поддержку.

### Решение

В проекте используется библиотека  
**@techlabteam/useful-hooks**  
как основной источник переиспользуемых React-хуков.

Документация библиотеки доступна на npm:
https://www.npmjs.com/package/@techlabteam/useful-hooks

### Последствия

- снижается дублирование логики
- поведение хуков становится единообразным
- развитие и исправления концентрируются в одном месте

## Barrel files (реэкспорт через `index.ts`)

### Контекст

Проект использует FSD-архитектуру с вложенной структурой директорий.
Прямые импорты из файлов приводят к длинным путям, усложняют рефакторинг
и размывают границы модулей.

### Принятое решение

Использовать **barrel files (`index.ts`)** как публичный API папок.
Импорты разрешены только через `index.ts`, без обращения к внутренним файлам.

### Причины

- сокращение и унификация импортов  
- явное определение публичных границ модуля  
- упрощение рефакторинга и перемещения файлов  
- повышение читаемости и предсказуемости кода  

### Последствия

- запрещены прямые импорты из внутренних файлов  
- `index.ts` является контрактом папки наружу  
- добавление публичных сущностей требует обновления barrel file
